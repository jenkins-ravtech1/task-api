חלק 1

למה CI/CD?

מה הבעיה שאוטומציה של בנייה ובדיקות פותרת — ולמה זה הדבר הראשון שמוסיפים אחרי שכותבים קוד.

מושג ליבה

מה זה CI/CD?

בפשטות

CI (אינטגרציה רציפה) = בכל פעם שמישהו דוחף קוד, מכונה אוטומטית בונה ובודקת אותו — כך תופסים שבירות תוך דקות, לא ימים. CD (אספקה רציפה) = ממשיכים אוטומטית גם לפריסה (מפגש 7).

לעומק טכני

CI מריץ pipeline על כל commit: checkout → build → test (→ lint, scan). מטרה: לשמור את ענף ה-main תמיד "ירוק" — תקין וניתן לשחרור בכל רגע.

היום מתמקדים ב-CI
נבנה את ה-build + test האוטומטי. ה-CD (פריסה אוטומטית) יתווסף במפגש 7, כשכבר יהיה לאן לפרוס.
הבעיה

בלי CI: באגים מתגלים מאוחר

  • מישהו דוחף קוד ששובר את הבנייה — ואף אחד לא יודע עד שמישהו אחר מנסה להריץ.
  • "בדקתי אצלי" — אבל שכחת קובץ, או הבדיקות לא רצו בכלל.
  • מיזוג של כמה אנשים יוצר התנגשויות שמתפוצצות רק בסוף.
עם CI

כל push נבנה ונבדק על מכונה נקייה. שבירה מסומנת באדום תוך דקות, עם לוג מדויק.

"מכונה נקייה"

ה-runner מתחיל מאפס בכל ריצה — אם זה עבד שם, זה לא תלוי ב"מה שמותקן אצלי". זה פותר את "אצלי זה עובד" עבור הבנייה והבדיקות.

חלק 2

עבודת צוות: Branches ו-PRs

CI חי על מנגנון ה-Pull Request של GitHub. נבין את זרימת העבודה שעליה הוא רץ.

מושג ליבה

Branches (ענפים)

בפשטות

branch הוא "קו עבודה" נפרד. עובדים על פיצ׳ר בענף משלו בלי לפגוע ב-main היציב, וממזגים פנימה כשמוכן.

לעומק טכני

branch הוא מצביע קל-משקל ל-commit. main הוא הענף הראשי. דפוס נפוץ: feature branch קצר, פותחים PR, וממזגים ל-main.

git checkout -b feature/health-message   # ענף חדש
# ...עורכים ומבצעים commit...
git push -u origin feature/health-message
מושג ליבה

Pull Request ו-Code Review

בפשטות

PR = בקשה למזג את הענף שלך ל-main. הוא מציג את ההבדלים (diff), מאפשר לחברי צוות להעיר (review), וכאן ה-CI רץ אוטומטית.

לעומק טכני

ה-PR מקשר ענף-מקור לענף-יעד; כל push לענף מעדכן את ה-PR ומריץ מחדש את ה-checks. מיזוג מתאפשר רק כשהבדיקות ירוקות והאישורים התקבלו.

feature branch→ Pull Request→ CI checks + review→ merge to main
שער איכות

Status Checks ו-Branch Protection

בפשטות

אפשר להגדיר ש-main "נעול": אי-אפשר למזג PR אם ה-CI אדום. כך קוד שבור פשוט לא נכנס.

לעומק טכני

Branch protection rule על main: דרישת PR, דרישת status check ירוק (ה-job שלנו), ואופציונלית אישור reviewer אחד לפחות.

נגדיר את זה היום
בסוף המפגש נסמן את ה-job של ה-CI כ-required על main — שער האיכות הראשון של הפרויקט.
חלק 3

אנטומיה של GitHub Actions

מנוע ה-CI/CD המובנה ב-GitHub. נבין את חלקיו לפני שנכתוב את ה-workflow הראשון.

מושג ליבה

מה זה GitHub Actions?

בפשטות

שירות שמריץ אוטומציה ישירות מתוך ה-repo: אתם מתארים בקובץ YAML "מתי" ו"מה לעשות", ו-GitHub מריץ את זה על מחשב זמני (runner).

לעומק טכני

runner = מכונה וירטואלית (למשל ubuntu-latest) שנוצרת לכל ריצה ונמחקת בסוף. GitHub מספק runners מנוהלים בחינם (מכסה חודשית) ל-repos ציבוריים ופרטיים.

היררכיה

Workflow → Job → Step

רכיבמה זה
Workflowקובץ YAML ב-.github/workflows/ — תהליך אוטומציה שלם, מופעל על אירוע
Jobקבוצת צעדים שרצה על runner אחד; jobs רצים במקביל כברירת מחדל
Stepפעולה בודדת בתוך job — או פקודת shell (run) או פעולה מוכנה (uses)
Actionרכיב לשימוש חוזר מה-Marketplace (למשל actions/checkout)
מתי זה רץ

Triggers (אירועים)

אירועמתי
pushבכל דחיפת קוד (לכל ענף או לענפים מסוימים)
pull_requestכשנפתח/מתעדכן PR — כאן נופלים שערי האיכות
workflow_dispatchהפעלה ידנית בלחיצת כפתור
scheduleלפי לוח זמנים (cron)
ה-CI שלנו
ירוץ על push (כל ענף) ועל pull_request ל-main — כך כל שינוי נבדק לפני מיזוג.
תחביר

YAML ב-60 שניות

  • הזחה (indentation) קובעת מבנה — רווחים, לא טאבים.
  • key: value — זוג מפתח-ערך.
  • - בתחילת שורה = פריט ברשימה.
  • # = הערה.
name: CI
on: [push]
jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - run: echo hello
הטעות הנפוצה ביותר
הזחה שגויה. שמרו על עקביות (2 רווחים לרמה), ו-VS Code יסמן שגיאות YAML.
שני סוגי צעדים

uses מול run

uses — פעולה מוכנה

- uses: actions/checkout@v4

מריץ רכיב מה-Marketplace. actions/checkout מושך את הקוד שלכם ל-runner.

run — פקודת shell

- run: mvn -B verify

מריץ פקודה רגילה ב-terminal של ה-runner — בדיוק מה שתריצו בעצמכם.

גרסאות actions
נצמיד גרסת major (למשל @v5). הגרסאות מתעדכנות — בדקו את האחרונה ב-GitHub Marketplace.
מהירות

Caching — לא להוריד הכול כל פעם

בפשטות

בלי cache, כל ריצה מורידה מחדש את כל תלויות ה-Maven — איטי. cache שומר אותן בין ריצות.

לעומק טכני

actions/setup-java מציע cache: maven מובנה — שומר את ~/.m2 עם מפתח לפי pom.xml. משחזר אוטומטית בריצה הבאה.

Secrets (טעימה)
ערכים רגישים (טוקנים, מפתחות) נשמרים ב-Settings → Secrets ונקראים כ-${{ secrets.NAME }} — לעולם לא בקוד. נשתמש בזה לעומק במפגש 7 (CD).
חלק 4

בונים את ה-CI של הפרויקט

עכשיו נכתוב את ci.yml שבונה ובודק את ה-Tasks API בכל push.

התוכנית

מה ה-CI שלנו יעשה

checkout→ setup Java 17→ mvn verify (build + tests)
  • רץ על push ועל pull_request ל-main.
  • מושך את הקוד, מתקין JDK 17 (עם cache), ומריץ mvn -B verify — קומפילציה + בדיקות יחידה.
  • נכשל אם הבנייה או בדיקה כלשהי נכשלת.
בהמשך
במפגש 3 נוסיף job שבונה גם את ה-Docker image; במפגש 5 יתווספו בדיקות אינטגרציה מול LocalStack.
קוד · ci.yml (1/2)

מתי לרוץ, ואיפה

.github/workflows/ci.yml 
name: CI
on:                          # מתי הצינור רץ
  push:
  pull_request:
    branches: [ main ]

jobs:
  build-test:
    runs-on: ubuntu-latest     # runner נקי בכל ריצה
    steps:
      ...                       # בשקופית הבאה
קוד · ci.yml (2/2)

הצעדים: checkout → Java → verify

.github/workflows/ci.yml (steps) 
    steps:
      - uses: actions/checkout@v4      # מושך את הקוד
      - uses: actions/setup-java@v4
        with:
          distribution: temurin
          java-version: '17'
          cache: maven                 # מאיץ ריצות
      - name: Build and test
        run: mvn -B verify             # קומפילציה + בדיקות

זהו — קובץ אחד, וכל push נבנה ונבדק אוטומטית.

הדגמה מוסיפים CI ל-repo
  1. גרסת הייחוס של היום (לראות את התוצאה המלאה): 
    git checkout v2-ci
  2. יוצרים את קובץ ה-workflow במיקום המדויק: 
    .github/workflows/ci.yml   # התוכן משתי השקופיות הקודמות
  3. דוחפים ל-GitHub: 
    git add .github
git commit -m "add CI workflow"
git push
  4. צופים בריצה בלשונית Actions של ה-repo — אמור להפוך לירוק.
הדגמה רואים את ה-CI תופס שבירה
  1. שוברים בכוונה — משנים ערך שבדיקה מצפה לו (למשל בתגובת /health).
  2. דוחפים ובודקים את לשונית Actions: 
    git commit -am "break health (demo)"
git push
# ה-CI נכשל — אדום, עם הלוג של הבדיקה שנפלה
  3. מתקנים ודוחפים שוב, וחוזר לירוק.
זה כל הרעיון
הבאג נתפס תוך דקות, אוטומטית, לפני שהגיע ל-main — ולא אצל משתמש.
הדגמה נועלים את main (Branch Protection)

בממשק GitHub הנוכחי:

  1. Repo → Settings → Branches → Add branch ruleset (או Add rule).
  2. מגדירים את היעד לענף main.
  3. מסמנים Require status checks to pass ובוחרים את ה-job שלנו (build-test).
  4. אופציונלי: Require a pull request before merging (+ אישור reviewer).
התוצאה
מעכשיו אי-אפשר למזג ל-main אם ה-CI אדום. שער איכות אמיתי.
מעבדה בנו את שלב 2 — 25 דקות
  1. צרו ענף: git checkout -b add-ci.
  2. הוסיפו את .github/workflows/ci.yml (מהשקופיות).
  3. דחפו ופתחו Pull Request ל-main; צפו ב-CI רץ על ה-PR.
  4. שברו בדיקה בכוונה, דחפו, וראו את ה-check הופך לאדום; תקנו וראו ירוק.
  5. אתגר: הפעילו Branch Protection שמחייב את ה-check לפני מיזוג, ואז מזגו את ה-PR.
סימן הצלחה
ה-PR מראה check ירוק, מיזוג נחסם כשהוא אדום, וה-main נשאר תקין.
משימת בית לפני מפגש 3
  1. הוסיפו בדיקת יחידה חדשה (למשל: POST עם כותרת ריקה מחזיר 400), ודאו שה-CI מריץ אותה.
  2. הוסיפו build badge ל-README.md (תג הסטטוס של ה-workflow) — כך רואים אם ה-build ירוק ישר מה-repo.
  3. קריאה: "פירמידת הבדיקות" (unit / integration / e2e) — נשתמש בה בהמשך.
טיפ ל-badge
ב-GitHub: Actions → בוחרים את ה-workflow → "Create status badge" → מדביקים ל-README.
