חלק 1

מ-CI ל-CD

בנינו ובדקנו אוטומטית (מפגש 2). עכשיו ניקח את זה עד הסוף — עד הפריסה לענן.

מושג ליבה

מה זה CD?

בפשטות

המשך טבעי של CI: אחרי שהקוד נבנה ונבדק, הצינור ממשיך אוטומטית גם לפרוס אותו לענן — בלי צעדים ידניים.

לעומק טכני

Continuous Delivery: כל commit שעובר מוכן לשחרור בלחיצה. Continuous Deployment: משוחרר אוטומטית. אנחנו נשלב — אוטומטי, עם שער אישור ל-production.

למה זה משנה
פחות טעויות אנוש, פריסות מהירות ועקביות, ויכולת לחזור אחורה בקלות. זה הלב של DevOps.
החיבור הגדול

כל מה שבנינו — בצינור אחד

push to main→ build image→ push ECR→ terraform apply→ deploy→ smoke test
  • CI (מפגש 2) · Docker (3) · ECR (6) · Terraform (6) — כולם כבר קיימים.
  • היום רק מחברים אותם ל-cd.yml אחד שרץ על כל push ל-main.
השאלה הראשונה
איך ה-pipeline מקבל הרשאה ל-AWS בלי לאחסן מפתחות סודיים? התשובה: OIDC.
חלק 2

אימות מאובטח: OIDC

איך לתת ל-GitHub Actions גישה ל-AWS — בלי מפתחות קבועים בכלל.

הבעיה

למה לא לשמור מפתחות גישה ב-CI?

  • מפתחות ארוכי-טווח דולפים: אם secret נחשף — לתוקף יש גישה לחשבון.
  • צריך לסובב אותם ידנית, ולעיתים שוכחים.
  • מפתח אחד = הרבה הרשאות = סיכון גדול.
הפתרון: OIDC

GitHub מוכיח ל-AWS "אני באמת ה-workflow הזה, מה-repo הזה" — ו-AWS נותן הרשאות זמניות שפגות מיד. אין מה לאחסן ואין מה לדלוף.

איך זה עובד

GitHub OIDC ↔ IAM Role

GitHub job— JWT חתום → AWS STS— בודק trust → הרשאות זמניות
בפשטות

ה-job מקבל "תעודת זהות" חתומה מ-GitHub, מציג אותה ל-AWS, ומקבל מפתחות זמניים לאותה ריצה בלבד.

לעומק טכני

AssumeRoleWithWebIdentity: ה-JWT נבדק מול OIDC provider רשום, וה-trust policy של ה-role מגבילה ל-repo ולענף מסוימים (repo:jenkins-ravtech1/task-api:ref:refs/heads/main).

הדגמה מגדירים את ה-OIDC role (ב-Terraform)
infra/iam.tf (תמצית) 
resource "aws_iam_openid_connect_provider" "github" {
  url             = "https://token.actions.githubusercontent.com"
  client_id_list  = ["sts.amazonaws.com"]
}
resource "aws_iam_role" "deploy" {
  # trust: רק ה-repo והענף שלנו רשאים לקחת את התפקיד
  # condition: repo:jenkins-ravtech1/task-api:ref:refs/heads/main
  # הרשאות: ECR push, terraform/state, SSM deploy
}
התוצאה
נוצר תפקיד שרק ה-workflow שלנו (מה-repo ומענף main) יכול "ללבוש" — בלי שום מפתח שמור.
חלק 3

בונים את צינור ה-CD

עכשיו כותבים את cd.yml — מחברים בנייה, דחיפה, apply ופריסה.

מושגים

Environments, אישורים, rollback

מושגמה זה
Environmentיעד פריסה (staging/production) עם כללים משלו
Approvalשער ידני — אדם מאשר לפני פריסה ל-production
Promotionקידום אותה גרסה מסביבה לסביבה
Rollbackחזרה לגרסה קודמת אם משהו נשבר
אצלנו
נגדיר environment בשם production שדורש אישור reviewer לפני שהפריסה רצה — בלם בטיחות אנושי.
קוד · cd.yml (1/3)

טריגר, הרשאות, ואימות OIDC

.github/workflows/cd.yml 
name: CD
on:
  push: { branches: [ main ] }
  workflow_dispatch:
permissions:
  id-token: write      # נדרש ל-OIDC
  contents: read
jobs:
  deploy:
    runs-on: ubuntu-latest
    environment: production   # דורש אישור
    steps:
      - uses: actions/checkout@v4
      - uses: aws-actions/configure-aws-credentials@v4
        with:
          role-to-assume: ${{ vars.AWS_DEPLOY_ROLE_ARN }}
          aws-region: ${{ vars.AWS_REGION }}
קוד · cd.yml (2/3)

בנייה ודחיפה ל-ECR + apply

.github/workflows/cd.yml (steps) 
      - uses: aws-actions/amazon-ecr-login@v2
      - name: Build and push image
        run: |
          docker build --platform linux/amd64 -f docker/Dockerfile -t $ECR_URI:$GITHUB_SHA .
          docker push $ECR_URI:$GITHUB_SHA
      - name: Terraform apply
        run: |
          terraform -chdir=infra init
          terraform -chdir=infra apply -auto-approve \
            -var image_tag=$GITHUB_SHA

אותו multi-stage build ו-Terraform ממפגש 6 — עכשיו רצים אוטומטית בצינור.

קוד · cd.yml (3/3)

פריסה ל-EC2 + smoke test

.github/workflows/cd.yml (steps) 
      - name: Deploy to EC2 (SSM)
        run: |
          PARAM=$(terraform -chdir=infra output -raw image_tag_parameter)
          INSTANCE_ID=$(terraform -chdir=infra output -raw instance_id)
          aws ssm put-parameter --name "$PARAM" --value "$GITHUB_SHA" --type String --overwrite
          aws ssm send-command \
            --instance-ids "$INSTANCE_ID" \
            --document-name AWS-RunShellScript \
            --parameters 'commands=["/usr/local/bin/deploy-app.sh"]'
      - name: Smoke test
        run: ./scripts/smoke-test.sh "$API_URL"   # נכשל אם /health אדום
למה SSM ולא SSH
פריסה דרך SSM לא דורשת מפתחות SSH בצינור — עקבי עם גישת ה-OIDC ללא מפתחות.
הדגמה פריסה אוטומטית מלאה
  1. גרסת הייחוס: git checkout v7-cd (כולל cd.yml וה-OIDC role).
  2. מגדירים repo variables (לא secrets!): AWS_DEPLOY_ROLE_ARN, AWS_REGION.
  3. דוחפים שינוי ל-main (או ממזגים PR) — ה-CD מתחיל.
  4. מאשרים את סביבת ה-production (שער האישור).
  5. צופים: בנייה → דחיפה ל-ECR → apply → פריסה → smoke test ירוק. השירות בענן מעודכן.
אימות
curl http://<api_url>/health מראה את הגרסה החדשה (APP_VERSION = ה-commit).
כשמשהו משתבש

Rollback — חזרה לאחור

בפשטות

אם פריסה חדשה שבורה — חוזרים מהר לגרסה הקודמת שעבדה, במקום לתקן בלחץ.

איך אצלנו

כל image מתויג ב-commit SHA. rollback = לפרוס מחדש tag קודם (להפעיל את ה-deploy עם SHA ישן). ה-smoke test שומר שלא נשאיר גרסה שבורה.

עיקרון
פריסות קטנות ותכופות = rollback קל. שינוי ענק = סיכון גדול.
מעבדה בנו את שלב 7 — 25 דקות
  1. ודאו שה-OIDC provider וה-deploy role קיימים (ב-Terraform).
  2. הגדירו AWS_DEPLOY_ROLE_ARN ו-AWS_REGION כ-repo variables.
  3. הוסיפו את .github/workflows/cd.yml.
  4. צרו environment בשם production עם required reviewer.
  5. מזגו PR ל-main, אשרו את הפריסה, וצפו בצינור עד ל-smoke test ירוק.
סימן הצלחה
push ל-main מפרס לבד את הגרסה החדשה לענן, אחרי אישור — בלי מפתחות קבועים בכלל.
משימת בית לפני מפגש 8
  1. תיעוד: כתבו ב-README איך הצינור עובד ואיך מפרסים (כולל ה-IaC).
  2. פריסה דרך PR: בצעו שינוי קטן, פתחו PR, מזגו, ואשרו פריסה — ראו את הגרסה משתנה.
  3. חשבו: השירות רץ בענן ומתעדכן לבד — אבל איך נדע אם הוא בריא, מהיר, או נופל? זה מפגש 8.
היגיינת עלויות
אם הקמתם תשתית לבדיקה — זכרו terraform destroy בסוף.
