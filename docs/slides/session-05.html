חלק 1

אחסון ומסדי נתונים

איפה שומרים קבצים, ואיפה שומרים נתונים מובנים — והבחירה לפרויקט שלנו.

אחסון אובייקטים

S3 — Simple Storage Service

בפשטות

"דיסק אינסופי" בענן לקבצים (אובייקטים): תמונות, גיבויים, קבצי לוג, ייצוא נתונים. מעלים לפי מפתח (שם), מקבלים URL.

לעומק טכני

אחסון אובייקטים עם עמידות גבוהה מאוד (11 nines). מאורגן ב-buckets; לא מערכת קבצים ולא מסד נתונים — מצוין לקבצים גדולים, לא לשאילתות.

בפרויקט שלנו
נשתמש ב-S3 להדגמה (למשל ייצוא/קבצים). הנתונים המובנים של המשימות יישבו ב-DynamoDB.
מסדי נתונים

RDS מול DynamoDB

RDS (רלציוני)DynamoDB (NoSQL)
מודלטבלאות, SQL, קשריםמפתח-ערך / מסמכים
סכמהקשיחה מראשגמישה
קנה מידהאנכי בעיקראופקי, מנוהל מלא
מתאים ל…קשרים מורכביםגישה מהירה לפי מפתח
למה DynamoDB לקורס
serverless, מנוהל מלא, on-demand (משלמים לפי שימוש), ומשתלב מצוין עם LocalStack — מושלם ל-Tasks API.
מידול בסיסי

איך נראית טבלת DynamoDB

בפשטות

טבלה = אוסף פריטים (items), לכל פריט מפתח ראשי (Partition Key) ייחודי. אצלנו: טבלה tasks עם מפתח id.

לעומק טכני

כל פריט הוא אוסף תכונות (attributes) — אין סכמה אחידה. הגישה היעילה היא לפי המפתח. תכנון המפתח קובע ביצועים (נלמד עוד בהמשך הקריירה).

tasks table:
{ "id": "a1b2", "title": "Learn Docker", "done": false }
{ "id": "c3d4", "title": "Deploy to AWS", "done": true  }

חלק 2

ארכיטקטורה מונחית-אירועים

למה לא לעשות הכול "כאן ועכשיו" בתוך הבקשה — ואיך תורים והודעות מפרידים בין רכיבים.

הבעיה

למה לפרק תלויות (decoupling)?

דמיינו ש-POST /tasks צריך גם לשלוח מייל, לעדכן דוח, ולקרוא לשירות חיצוני — הכול בתוך הבקשה.

  • המשתמש מחכה לכל הפעולות — איטי.
  • אם שירות אחד נופל — כל הבקשה נכשלת.
  • קשה להוסיף "עוד מאזין" בלי לגעת בקוד ה-API.
הפתרון

ה-API רק מפרסם אירוע ("נוצרה משימה") לתור, ועונה מיד. עיבוד נוסף קורה אסינכרונית ברקע — מהיר, עמיד, וקל להרחבה.

מושג ליבה

SQS — תור הודעות

בפשטות

"תיבת דואר" בין רכיבים: המפרסם שם הודעה בתור, והצרכן מושך אותה כשהוא פנוי. אם הצרכן עסוק או נפל — ההודעה ממתינה, לא הולכת לאיבוד.

לעומק טכני

תור מנוהל, at-least-once delivery. הודעה שנקראה "מוסתרת" (visibility timeout) עד שתימחק; אם לא עובדה — חוזרת. הודעות בעייתיות עוברות ל-DLQ (תור מתים).

מושג ליבה

SNS — פרסום והרשמה

בפשטות

"רמקול": מפרסמים הודעה ל-topic, וכל מי שנרשם (מייל, SMS, תור אחר) מקבל אותה. אחד-לרבים (fan-out).

SQS מול SNS

SQS = הודעה לצרכן אחד שמושך בקצב שלו. SNS = שידור לכל המנויים מיד (push). לרוב משלבים: SNS מפזר ל-SQS.

הארכיטקטורה שלנו

הזרימה: API → SQS → Lambda → SNS

POST /tasks→ SQS (תור)→ Lambda (צרכן)→ SNS (התראה)

ה-API שומר את המשימה ומפרסם אירוע לתור; ה-Lambda צורך מהתור ומפרסם התראה ב-SNS — בדיוק הדיאגרמה מהסילבוס.

Idempotency (חשוב!)
SQS עלול למסור הודעה יותר מפעם אחת. הצרכן חייב להיות אידמפוטנטי — לעבד את אותו taskId פעמיים בלי תופעות לוואי כפולות.
חלק 3

LocalStack

להריץ את AWS על המחשב — לפתח ולתרגל DynamoDB/SQS/SNS בלי לגעת בענן ובלי לשלם.

מושג ליבה

מה זה LocalStack?

בפשטות

חיקוי של AWS שרץ ב-container על המחשב שלכם. מדברים אליו עם אותן פקודות AWS בדיוק — רק שמפנים אותן אליו, לא לענן.

לעומק טכני

מממש את ה-APIs של עשרות שירותי AWS מאחורי "שער" אחד על פורט 4566. מפנים אליו עם --endpoint-url או דרך awslocal.

למה זה מושלם לקורס
מתרגלים DynamoDB/SQS/SNS ללא עלות וללא סיכון — בדיוק ה"מקומי" בשילוב "אמיתי" + "מקומי" של הקורס.
קוד · compose

מוסיפים את LocalStack ל-compose

docker/docker-compose.yml (תוספת) 
services:
  app:
    # ...כמו מפגש 3, אבל עכשיו:
    environment:
      - STORAGE=dynamodb
      - AWS_ENDPOINT_URL=http://localstack:4566
      - TASKS_TABLE=tasks
      - TASK_EVENTS_QUEUE_URL=http://localstack:4566/000000000000/task-events
    depends_on: [ localstack ]
  localstack:
    image: localstack/localstack
    ports: [ "4566:4566" ]
    environment: [ "SERVICES=dynamodb,sqs,sns,lambda,s3" ]

זוכרים את ההבטחה ממפגש 3? עכשיו ה-compose מריץ שני שירותים יחד.

הדגמה יוצרים טבלה, תור ונושא ב-LocalStack
  1. מעלים את LocalStack (כחלק מה-compose או לבד).
  2. טבלת DynamoDB (awslocal = aws מול LocalStack): 
    awslocal dynamodb create-table --table-name tasks \
  --attribute-definitions AttributeName=id,AttributeType=S \
  --key-schema AttributeName=id,KeyType=HASH \
  --billing-mode PAY_PER_REQUEST
  3. תור SQS ונושא SNS: 
    awslocal sqs create-queue --queue-name task-events
awslocal sns create-topic --name task-notifications
  4. בודקים שנוצרו: awslocal dynamodb list-tables.
ב-repo
סקריפט ב-localstack/init/ יוצר את כל אלה אוטומטית בעליית ה-compose.
Windows
פקודות רב-שורתיות (\) רצות ב-WSL2/Git Bash; ב-PowerShell השתמשו ב-backtick או בשורה אחת (ראו מפגש 1).

מימוש DynamoDbTaskRepository

הקסם של ההפשטה
בוחרים מימוש לפי STORAGE=dynamodb. אותו endpointOverride מצביע ל-LocalStack בפיתוח ול-AWS אמיתי בענן.
קוד · פרסום אירוע

פרסום ל-SQS ב-POST

SqsEventPublisher.java 
// אחרי שמירת המשימה, מפרסמים אירוע — בלי לעכב את התשובה
String body = "{\"type\":\"TASK_CREATED\","
            + "\"taskId\":\"" + task.id() + "\"}";
sqs.sendMessage(b -> b.queueUrl(queueUrl).messageBody(body));
חשוב

כשל בפרסום לא מפיל את הבקשה — רק מתעד אזהרה. המשתמש כבר קיבל 201; העיבוד הנוסף אסינכרוני.

קוד · הצרכן

ה-Lambda שצורך מהתור

lambda/.../Handler.java 
public class Handler implements RequestHandler<SQSEvent, Void> {
  public Void handleRequest(SQSEvent e, Context ctx) {
    for (var msg : e.getRecords()) {
      // 1) לפענח את ההודעה (taskId)
      // 2) לעבד (אידמפוטנטי!)
      sns.publish(b -> b.topicArn(topic)
          .message("Task created: " + taskId));   // 3) להתריע
    }
    return null;
  }
}

חיבור התור ל-Lambda (event source mapping) יוגדר ב-Terraform במפגש 6; ב-LocalStack נחבר אותו לתרגול.

הדגמה הזרימה המלאה מקצה לקצה
  1. גרסת הייחוס: git checkout v5-data-events, ואז מעלים הכול: 
    docker compose -f docker/docker-compose.yml up --build
  2. יוצרים משימה (ה-API שומר ב-DynamoDB ומפרסם ל-SQS): 
    curl -X POST localhost:8080/tasks \
  -H "Content-Type: application/json" -d '{"title":"Ship it"}'
  3. מאמתים שנשמר ב-DynamoDB: 
    awslocal dynamodb scan --table-name tasks   # המשימה שם
  4. צופים שה-Lambda צרך את ההודעה ופרסם ל-SNS (בלוגים).

מעבדה בנו את שלב 5 — 25 דקות
  1. העלו LocalStack וצרו טבלה, תור ונושא (awslocal).
  2. הריצו את האפליקציה עם STORAGE=dynamodb ו-AWS_ENDPOINT_URL ל-LocalStack.

סימן הצלחה
המשימות שורדות הפעלה מחדש (נשמרות ב-DynamoDB), והאירועים זורמים API → SQS → Lambda → SNS.
