היום — שלב 6: IaC + ECR

אורזים image ל-production, דוחפים ל-ECR, ומקימים את כל מה שעשינו ביד (מפגשים 4–5) כקוד עם Terraform.

כבר מאחורינו · הבא

מפגשים 4–5: הקמנו ידנית EC2, DynamoDB, תור ונושא. מפגש 7: נחבר את הכול ל-CD — פריסה אוטומטית בלחיצת push.

חלק 1

image רזה ו-ECR

נשפר את הדרך שבה אנחנו בונים ומאחסנים את ה-image, לקראת הרצה בענן.

השיפור

הבעיה עם ה-Dockerfile ממפגש 3

  • שני צעדים: צריך mvn package לפני docker build.
  • image גדול: כלל בסיס JRE + ה-JAR, אבל לא היה ממוטב.
  • תלות בסביבה המקומית לבניית ה-JAR.
הפתרון: multi-stage build

בונים את ה-JAR בתוך שלב בנייה (עם Maven), ומעתיקים רק את התוצר לשלב הרצה רזה. docker build אחד עושה הכול — קטן ובטוח יותר.

קוד · multi-stage

ה-Dockerfile הרב-שלבי

docker/Dockerfile 
# שלב 1: בנייה (כולל Maven ו-JDK)
FROM maven:3.9-eclipse-temurin-17 AS build
WORKDIR /src
COPY . .
RUN mvn -q -pl app -am -DskipTests package

# שלב 2: הרצה (JRE רזה בלבד)
FROM eclipse-temurin:17-jre-alpine AS runtime
WORKDIR /app
COPY --from=build /src/app/target/*-shaded.jar app.jar
EXPOSE 8080
ENTRYPOINT ["java","-jar","app.jar"]
הרווח
שלב הבנייה (הכבד) נזרק; ב-image הסופי יש רק JRE + ה-JAR. docker build יחיד, בלי mvn ידני.
מושג ליבה

Amazon ECR — registry פרטי

בפשטות

כמו Docker Hub, אבל פרטי ובתוך AWS. כאן יישב ה-image שהענן ימשוך ויריץ. הרשאות נשלטות דרך IAM.

אימות

מתחברים עם טוקן זמני:

aws ecr get-login-password --region <region> \
 | docker login --username AWS \
   --password-stdin <acct>.dkr.ecr.<region>.amazonaws.com
הדגמה בונים רזה ודוחפים ל-ECR
  1. בונים את ה-image הרב-שלבי (פקודה אחת, בלי mvn מקומי): 
    docker build --platform linux/amd64 -f docker/Dockerfile -t tasks-api .
  2. מתחברים ל-ECR (ה-repo נוצר ב-Terraform בהמשך): 
    aws ecr get-login-password --region <region> \
 | docker login --username AWS --password-stdin <repo-uri>
  3. מתייגים ודוחפים: 
    docker tag tasks-api <repo-uri>:1.0
docker push <repo-uri>:1.0
השוואה למפגש 3
שם דחפנו ל-Docker Hub ציבורי; כאן ל-ECR פרטי בתוך החשבון, ממנו השרת בענן ימשוך.
Windows
פקודות רב-שורתיות (\) רצות ב-WSL2/Git Bash; ב-PowerShell השתמשו ב-backtick או בשורה אחת (ראו מפגש 1).
חלק 2

תשתית כקוד (IaC)

להפסיק "ללחוץ בקונסולה" — ולתאר את כל התשתית בקבצים, כמו קוד.

הבעיה

מה רע ב"הקלקה בקונסולה"?

  • לא ניתן לשחזור: מי זוכר את כל ה-30 קליקים שעשינו במפגשים 4–5?
  • לא עקבי: סביבת dev ו-prod יוצאות שונות ("drift").
  • לא ניתן לסקירה: אין diff, אין code review, אין היסטוריה.
  • קשה למחוק: שכחתם משאב דולק = חיוב מתמשך.
הפתרון

מתארים את התשתית בקוד, שומרים ב-Git, ומקימים/מוחקים אותה בפקודה — ניתן לשחזור, עקבי, ונבדק.

מושג ליבה

מה זה Infrastructure as Code?

בפשטות

כותבים קובץ שאומר "אני רוצה שרת, טבלה, תור ונושא", ומריצים פקודה שמגשימה את זה ב-AWS. אם משנים את הקובץ — התשתית מתעדכנת בהתאם.

לעומק טכני

הצהרתי (declarative): מתארים את מצב היעד, לא את הצעדים. idempotent: הרצה חוזרת לא יוצרת כפילויות — מביאה למצב הרצוי. הכלי מחשב את ה-diff ומיישם רק אותו.

הכלי שלנו

Terraform — מושגי יסוד

מושגמה זה
Providerתוסף שמדבר עם הספק (כאן: AWS)
Resourceמשאב שרוצים להקים (EC2, טבלה, תור...)
Variableפרמטר (region, גודל שרת...) — להתאמה בלי לשנות קוד
Stateקובץ שזוכר מה Terraform כבר יצר במציאות
למה Terraform
סטנדרט תעשייתי, קריא, ועובד מול ספקים רבים. (חלופות נטיביות ל-AWS: CloudFormation, CDK.)
מחזור העבודה

init → plan → apply → destroy

פקודהמה היא עושה
terraform initמוריד את ה-provider ומכין את התיקייה
terraform planמציג תצוגה מקדימה של מה ייווצר/ישתנה/יימחק
terraform applyמיישם בפועל ב-AWS
terraform destroyמוחק את כל מה שנוצר — היגיינת עלויות
תמיד plan לפני apply
קוראים את התצוגה המקדימה לפני שמאשרים — כך לא מופתעים ממחיקה או שינוי לא מכוון.
חלק 3

מקימים את מחסנית הענן בקוד

נכתוב Terraform שמקים את כל מה שהפרויקט צריך — בפקודה אחת.

מבנה

קבצי ה-Terraform שלנו

infra/ 
infra/
  providers.tf   # הספק (AWS) + region
  variables.tf   # פרמטרים
  ecr.tf         # registry ל-image
  ec2.tf         # השרת + user-data (מושך מ-ECR ומריץ)
  dynamodb.tf    # טבלת המשימות
  messaging.tf   # SQS + SNS
  lambda.tf      # הצרכן + חיבור לתור
  iam.tf         # תפקידים והרשאות
  outputs.tf     # כתובת ה-API וכו'

כל מה שהקמנו ביד במפגשים 4–5 — עכשיו כקוד, ב-Git.

קוד · provider + resource

הספק ומשאב ראשון

infra/providers.tf + ecr.tf 
terraform {
  required_providers {
    aws = { source = "hashicorp/aws", version = "~> 5.0" }
  }
}
provider "aws" { region = var.aws_region }

resource "aws_ecr_repository" "app" {
  name = "tasks-api"
}
תבנית קבועה
כל משאב: resource "סוג" "שם" { ...הגדרות... }. בדקו תמיד את גרסת ה-major העדכנית של ה-provider ב-Terraform Registry.
קוד · נתונים ואירועים

טבלה, תור ונושא — כקוד

infra/dynamodb.tf + messaging.tf 
resource "aws_dynamodb_table" "tasks" {
  name         = "tasks"
  billing_mode = "PAY_PER_REQUEST"
  hash_key     = "id"
  attribute { name = "id"  type = "S" }
}

resource "aws_sqs_queue" "events" { name = "task-events" }
resource "aws_sns_topic"  "notify" { name = "task-notifications" }

אותם משאבים שיצרנו ב-awslocal מקומית — עכשיו אמיתיים בענן, מתוארים בקוד.

קוד · השרת

שרת שמריץ את ה-image מ-ECR

infra/ec2.tf 
resource "aws_instance" "app" {
  ami                  = data.aws_ssm_parameter.al2023.value
  instance_type        = var.instance_type
  iam_instance_profile = aws_iam_instance_profile.app.name
  vpc_security_group_ids = [aws_security_group.app.id]
  user_data = templatefile("${path.module}/user-data.sh.tftpl", { ... })  # מתקין docker, מושך מ-ECR, מריץ
}
user-data
סקריפט שרץ באתחול השרת: מתקין Docker, מתחבר ל-ECR, מושך את ה-image ומריץ אותו כשירות. ה-IAM role נותן לשרת הרשאה ל-ECR/DynamoDB — בלי מפתחות.
הדגמה מקימים את הכול בפקודה
  1. גרסת הייחוס: git checkout v6-iac-ecr, ואז לתיקיית infra/.
  2. אתחול ותצוגה מקדימה: 
    terraform init
terraform plan      # מה ייווצר?
  3. הקמה: 
    terraform apply     # מאשרים -> נוצרים ECR, EC2, DynamoDB, SQS, SNS, Lambda, IAM
  4. דוחפים את ה-image ל-ECR שנוצר, והשרת מריץ אותו.
  5. מאמתים דרך הפלט (terraform output): 
    curl http://<api_url>/health
הדגמה ניקוי + מושג ה-state
  1. מוחקים הכול בפקודה אחת (היגיינת עלויות): 
    terraform destroy
  2. איך Terraform יודע מה למחוק? מקובץ ה-state — שמתעד מה נוצר במציאות.
state בפשטות

"רשימת המלאי" של מה ש-Terraform יצר. בלעדיה הוא לא יודע מה כבר קיים.

remote backend

בצוות שומרים את ה-state מרוחק (S3 + נעילה ב-DynamoDB) כדי שכולם יעבדו על אותו מצב. (קריאה בבית.)

מעבדה בנו את שלב 6 — 25 דקות
  1. בנו את ה-image הרב-שלבי (docker build אחד).
  2. הריצו terraform init ו-terraform plan בתיקיית infra/, וקראו את התצוגה המקדימה.
  3. הקימו עם terraform apply; דחפו את ה-image ל-ECR שנוצר.
  4. גשו ל-http://<api_url>/health מהפלט; צרו משימה ובדקו שנשמרה ב-DynamoDB האמיתי.
  5. חשוב מאוד: סיימו ב-terraform destroy.
סימן הצלחה + זהירות
ה-API רץ בענן מתשתית שהוקמה כקוד — ואז נמחק לגמרי ב-destroy כדי לא לצבור עלות.
משימת בית לפני מפגש 7
  1. פרמטריזציה: הוציאו ערכים קשיחים ל-variables.tf (aws_region, instance_type) והשתמשו ב-var..
  2. קריאה: Terraform remote state ו-backends (S3 + נעילה) — למה צוותים צריכים את זה.
  3. חשבו: הקמנו ידנית; דחפנו image ידנית. מה אם push ל-main היה עושה הכול לבד? זה מפגש 7.
ודאו
ש-terraform destroy רץ — שלא יישארו משאבים דולקים.
סיכום מפגש 6

מה בנינו, ולאן ממשיכים

שלב 6 הושלם — IaC + ECR

  • בנינו image רזה (multi-stage) ודחפנו ל-ECR.
  • הקמנו את כל מחסנית הענן עם Terraform.
  • מחקנו הכול ב-destroy — תשתית חד-פעמית וניתנת לשחזור.

הבא — מפגש 7: CD

  • נחבר CI + Docker + ECR + Terraform לצינור אחד.
  • push ל-main מפעיל בנייה, דחיפה, apply ופריסה — אוטומטית.
לא לשכוח
שהתשתית נמחקה, וה-Terraform מעודכן ב-repo.
