חלק 1

חשבון בטוח ועלויות בשליטה

לפני שנוגעים במשהו — מגדירים רשת ביטחון כספית. בענן, טעות יכולה לעלות כסף.

התמצאות

הקונסולה ומודל החיוב

בפשטות

הקונסולה היא ממשק הניהול של AWS. שני דברים לשים לב אליהם תמיד: באיזה Region אתם (פינה עליונה), וכמה אתם מוציאים (Billing).

מודל החיוב (2025+)

Free Plan חדש: עד $200 קרדיט, בתוקף 6 חודשים או עד שייגמר — המוקדם מביניהם. עוצר לפני חיוב. קיימים גם 30+ שירותים תמיד‑חינמיים.

החוק הראשון בענן
כל משאב שמדליקים — צריך לדעת לכבות. נסיים כל מפגש בכיבוי/מחיקה, ונתחיל היום בהגדרת התראות תקציב.
הדגמה רשת ביטחון: Budget + התראה
  1. בקונסולה: Billing and Cost Management → Budgets → Create budget.
  2. בוחרים Cost budget, תקופה חודשית, וסכום (למשל $10).
  3. מוסיפים alert threshold (למשל 80%) עם האימייל שלכם.
  4. שומרים — תקבלו מייל אם מתקרבים לסכום.
למה דווקא קודם
זו ההרגלה הכי חשובה: לפני שמדליקים משאבים, מוודאים שיש מי "שיצעק" אם העלות מטפסת.
חלק 2

IAM: מי מורשה לעשות מה

שירות ההרשאות של AWS. הבסיס לאבטחה — והצד שלכם במודל האחריות המשותפת.

מושג ליבה

אבני הבניין של IAM

רכיבמה זה
Userזהות לאדם או אפליקציה
Groupאוסף משתמשים עם אותן הרשאות
Policyמסמך JSON שמגדיר מה מותר (פעולות על משאבים)
Roleזהות עם הרשאות שנלקחת זמנית — למשל ע"י EC2 או GitHub
ההבדל החשוב

User = מפתחות קבועים. Role = הרשאות זמניות, בלי מפתחות לשמור. תפקידים עדיפים לכל מה שאינו אדם.

עקרונות ופרקטיקה עדכנית

הרשאה מינימלית, MFA, ותפקידים

  • הרשאה מינימלית (least privilege): נותנים רק את מה שצריך, לא יותר.
  • MFA: אימות דו-שלבי — חובה ל-root ולמשתמשים.
  • לא משתמשים ב-root לעבודה יומיומית.
הפרקטיקה העדכנית (2026)

לצוותים אמיתיים AWS ממליצה על IAM Identity Center לגישת אנשים (הרשאות זמניות), ועל Roles לעומסים. בקורס בחשבון יחיד נשתמש במשתמש IAM עם MFA לנוחות — אבל לעומסים (EC2 בהמשך, ו-CI/CD במפגש 7) נשתמש ב-Roles ללא מפתחות קבועים. זה החלק המודרני שחשוב להפנים.

הדגמה מאבטחים את החשבון
  1. מפעילים MFA ל-root: Account menu → Security credentials → Assign MFA (אפליקציית authenticator).
  2. יוצרים משתמש IAM: IAM → Users → Create user, עם גישה לקונסולה.
  3. מצרפים הרשאות דרך Group (למשל מדיניות ניהול לקורס), ומפעילים MFA גם למשתמש.
  4. מתנתקים מ-root ונכנסים מחדש כמשתמש ה-IAM — מכאן עובדים.
זהירות עם מפתחות
אם יוצרים access keys — לעולם לא להעלות ל-Git. בקורס נעדיף Roles ו-OIDC במקום מפתחות.
חלק 3

EC2: שרת בענן

מכונה וירטואלית שאפשר להקים בדקות. כאן נריץ את ה-container שלנו.

מושג ליבה

מה זה EC2?

בפשטות

מחשב וירטואלי בענן שאתם שוכרים לפי שעה. בוחרים מערכת הפעלה וגודל, ומקבלים שרת תוך דקות עם כתובת IP ציבורית.

מושגי מפתח

AMI = תבנית מערכת ההפעלה (נשתמש ב-Amazon Linux 2023). instance type = גודל (t3.micro — חסכוני). key pair = מפתח להתחברות. Security Group = חומת אש.

רשת ואבטחה

Security Groups — חומת האש של השרת

בפשטות

קובעת איזו תעבורה נכנסת לשרת. כברירת מחדל הכול חסום — צריך לפתוח במפורש את הפורטים שצריך.

לפרויקט שלנו

נפתח 22 (SSH/Instance Connect) ו-8080 (ה-Tasks API). כלל: מקור (CIDR), פרוטוקול, ופורט. לצמצום סיכון — להגביל ל-IP שלכם.

טעות נפוצה
"השרת רץ אבל אי-אפשר לגשת" — כמעט תמיד Security Group שלא פתח את הפורט. זה הדבר הראשון לבדוק.
הדגמה מקימים שרת EC2

אשף ההקמה הנוכחי הוא עמוד אחד:

  1. EC2 → Launch instance; נותנים שם (tasks-api).
  2. AMI: Amazon Linux 2023 (מסומן Free tier eligible).
  3. Instance type: t3.micro (או t2.micro).
  4. Key pair: יוצרים אחד (או "proceed without" ונשתמש ב-Instance Connect).
  5. Network settings: ב-Security Group מאפשרים SSH ומוסיפים Custom TCP 8080.
  6. Launch instance, וממתינים שהמצב יהיה Running.
הדגמה מריצים את ה-image על השרת
  1. מתחברים: בוחרים את ה-instance → Connect → EC2 Instance Connect → Connect (טרמינל בדפדפן).
  2. מתקינים Docker ומפעילים: 
    sudo dnf install -y docker
sudo systemctl enable --now docker
  3. מריצים את ה-image שדחפנו ל-Docker Hub במפגש 3: 
    sudo docker run -d -p 8080:8080 <user>/tasks-api:1.0
  4. ניגשים מהדפדפן/מהמחשב דרך ה-IP הציבורי של השרת: 
    curl http://<public-ip>:8080/health
רגע של ניצחון
אותו image בדיוק שרץ על המחשב שלכם — רץ עכשיו בענן, נגיש מכל מקום.
חלק 4

Lambda: מחשוב serverless

הקצה השני של ספקטרום המחשוב — קוד שרץ בלי שרת לנהל.

מושג ליבה

מה זה Lambda?

בפשטות

מעלים פונקציה, ו-AWS מריץ אותה כש"קורה משהו" (אירוע) — בלי שרת שאתם מתחזקים. משלמים רק על זמן הריצה בפועל.

לעומק טכני

FaaS מונחה‑אירועים: AWS מקצה runtime לבקשה, מריץ את ה-handler, ומכבה. מתאים לעיבוד אסינכרוני וזרימות אירועים — בדיוק מה שנבנה במפגש 5 (SQS → Lambda).

מתי כל אחד

EC2 מול Lambda

EC2Lambda
מודלשרת שרץ תמידפונקציה לפי אירוע
ניהולאתם (OS, עדכונים)AWS מנהל
תמחורלפי זמן הדלקהלפי קריאה/זמן ריצה

בפרויקט שלנו
ה-Tasks API רץ על EC2 (שירות רציף); Lambda יעבד אירועי "נוצרה משימה" מהתור (מפגש 5).
הדגמה פונקציית Lambda ראשונה
  1. Lambda → Create function → Author from scratch; שם + Runtime: Java 17.
  2. מעלים את ה-handler המוכן מה-repo (lambda/) — או מתחילים מדוגמה.
  3. יוצרים Test event פשוט ולוחצים Test.
  4. צופים בתוצאה ובלוגים ב-CloudWatch — הפונקציה רצה בלי שום שרת שהקמנו.
זו רק היכרות
במפגש 5 נחבר Lambda אמיתי לתור SQS שמקבל אירועים מה-API — הצרכן בארכיטקטורה.
מעבדה בנו את שלב 4 — 25 דקות
  1. הגדירו Budget + התראת אימייל.
  2. ודאו MFA ל-root וצרו משתמש IAM לעבודה.
  3. הקימו EC2 (Amazon Linux 2023, t3.micro) עם SG שפותח 22 ו-8080.
  4. התחברו, התקינו Docker, והריצו את ה-image שלכם מ-Docker Hub.
  5. גשו ל-http://<public-ip>:8080/health מהדפדפן.
  6. אתגר: צרו פונקציית Lambda והריצו Test event.
סימן הצלחה
ה-API עונה דרך ה-IP הציבורי, וה-Lambda מחזיר תוצאה ב-Test.
משימת בית לפני מפגש 5
  1. כבו/מחקו את ה-EC2! (Stop או Terminate) — היגיינת עלויות. ודאו שאין משאבים שרצים לחינם.
  2. תרגיל AWS CLI: התקינו והגדירו, ונסו aws sts get-caller-identity ו-aws s3 ls.
  3. קריאה: מהו מסד נתונים NoSQL (DynamoDB) — לקראת מפגש 5.
חשוב
שרת EC2 שנשאר דולק צורך קרדיט. בדקו את ה-Budget שיצרתם — הוא שם בדיוק בשביל זה.
