מוסיפים ניטור לשירות הפרוס: לוגים, מטריקות והתחקות — וסוגרים את מחזור החיים Code → Observe בפרויקט שלם.

מאחורינו

חלק 1

למה Observability?

שירות בענן הוא "קופסה שחורה" — בלי ניטור, נגלה תקלות מהמשתמשים. נפתח את הקופסה.

מושג ליבה

Monitoring מול Observability

בפשטות

Monitoring = לעקוב אחרי דברים שאנחנו יודעים לבדוק מראש ("האם השרת חי?"). Observability = יכולת לחקור ולהבין גם תקלות שלא צפינו, מתוך הנתונים שהמערכת פולטת.

לעומק טכני

Observability נשענת על שלושה סוגי טלמטריה ("שלושת העמודים"): Logs (מה קרה), Metrics (כמה), ו-Traces (איפה בזרימה). יחד הם מאפשרים לשאול שאלות חדשות בלי לשנות קוד.

המסגרת

שלושת העמודים

Logs

רישום אירועים בזמן — "מה קרה ומתי". טקסט/JSON לכל בקשה או שגיאה.

Metrics

מספרים לאורך זמן — "כמה": בקשות לשנייה, אחוז שגיאות, latency.

Traces

מסע של בקשה בודדת בין רכיבים — "איפה" הזמן/התקלה בזרימה.

ב-AWS
CloudWatch מרכז Logs ו-Metrics; X-Ray מציג Traces. כולם מוזנים מהשירות שלנו.
חלק 2

עמוד 1: Logs

הדרך הפשוטה והחשובה ביותר לדעת מה קרה — אם כותבים אותם נכון.

מושג ליבה

לוגים מובנים (Structured Logging)

בפשטות

במקום שורות טקסט חופשי, כותבים כל לוג כ-JSON עם שדות. אז אפשר לסנן ולחפש לפי שדה — "כל הבקשות עם status 500".

בפרויקט שלנו

כל בקשה כותבת שורה אחת:

{"ts":"...","requestId":"a1b2",
 "method":"POST","path":"/tasks",
 "status":201,"durationMs":12}
איסוף ותשאול

CloudWatch Logs ו-Logs Insights

בפשטות

השירות שולח את הלוגים ל-CloudWatch, ושם אפשר לחפש בהם — גם של ה-EC2 וגם של ה-Lambda, במקום אחד.

Logs Insights

שפת שאילתות על הלוגים:

fields @timestamp, status, durationMs
| filter status >= 500
| sort @timestamp desc
איך הלוגים מגיעים
הקונטיינר על EC2 שולח stdout ל-CloudWatch (דרך log driver); Lambda שולח אוטומטית. הכול הוגדר ב-Terraform (infra/observability.tf, מפגש 8).
הדגמה שולחים ומתשאלים לוגים
  1. מייצרים תעבורה — כמה בקשות ל-API הפרוס (כולל אחת שגויה).
  2. פותחים CloudWatch → Log groups ובוחרים את ה-log group של השירות.
  3. מריצים שאילתה ב-Logs Insights שמסננת status >= 500.
  4. מאתרים את הבקשה הבעייתית לפי requestId ורואים את כל ההקשר.
הערך
מ"משהו נשבר" ל"הבקשה הזו, בזמן הזה, עם השגיאה הזו" — תוך שניות.
חלק 3

עמוד 2: Metrics ו-Alarms

מספרים לאורך זמן — ומישהו ש"יצעק" כשמשהו חורג.

מושג ליבה

Metrics — מדידה לאורך זמן

בפשטות

ערכים מספריים שנמדדים שוב ושוב: כמה בקשות, כמה שגיאות, כמה זמן לקחה תגובה. מציגים אותם בגרפים (dashboard).

RED / USE

מסגרות נפוצות: RED — Rate, Errors, Duration (לשירותים). USE — Utilization, Saturation, Errors (למשאבים).

בפרויקט שלנו
האפליקציה פולטת metric מותאם (TasksCreated) ו-durationMs (latency) — דרך EMF (הלוגים המובנים הופכים גם למטריקות).
תגובה אוטומטית

Dashboards ו-Alarms

בפשטות

Dashboard = מסך גרפים לתמונת מצב. Alarm = חוק על metric ("אם יש שגיאות 5xx ל-5 דקות") ששולח התראה.

בפרויקט שלנו

CloudWatch Alarm על 5xx או על שגיאות Lambda מפעיל פרסום ל-SNS (אותו נושא מהארכיטקטורה), ומשם מייל. הוגדר ב-Terraform.

Metric→ Alarm (חורג מסף)→ SNS→ מייל לצוות
הדגמה metric, dashboard ו-alarm
  1. צופים ב-metric TasksCreated ו-durationMs ב-CloudWatch → Metrics.
  2. מוסיפים ל-dashboard גרף של בקשות ושגיאות.
  3. יוצרים alarm: 5xx > 0 ל-5 דקות, והפעולה: SNS.
  4. מפעילים שגיאה בכוונה (בקשה לא תקינה חוזרת) — וצופים ב-alarm עובר ל-ALARM ובמייל שמגיע.
סגירת מעגל
אותו SNS שבנינו במפגש 5 לאירועי משימות — משמש עכשיו גם להתראות תקלה.
חלק 4

עמוד 3: Traces

לעקוב אחרי בקשה בודדת לאורך כל הרכיבים — API → SQS → Lambda.

מושג ליבה

Distributed Tracing

בפשטות

כשבקשה עוברת בין כמה שירותים, trace מחבר את כל התחנות לסיפור אחד — ומראה איפה הזמן הלך או איפה נשבר.

לעומק טכני

trace מורכב מ-spans (מקטעי עבודה) עם מזהה משותף (trace id) שעובר בין הרכיבים. כך רואים API → SQS → Lambda כשרשרת אחת.

הגישה העדכנית

OpenTelemetry → X-Ray

בפשטות

מחברים "סוכן" (agent) לאפליקציה בלי לשנות קוד, והוא אוסף traces ושולח אותם ל-X-Ray לצפייה.

לעומק טכני

ADOT (חבילת OpenTelemetry של AWS) — סוכן Java שמתחבר עם -javaagent ומבצע auto-instrumentation ל-HTTP ולקריאות AWS SDK.

# מחברים את הסוכן — אפס שינויי קוד
java -javaagent:/opt/aws-opentelemetry-agent.jar -jar app.jar
OTEL_TRACES_EXPORTER=otlp        # → אוסף ADOT → X-Ray
עדכני, לא מיושן
AWS ממליצה היום על OpenTelemetry, וה-X-Ray SDK הישן בדרך ליציאה משימוש. בונוס: אותו סוכן מפעיל גם CloudWatch Application Signals (APM) בהגדרה אחת.
הדגמה trace מקצה לקצה ב-X-Ray
  1. מוודאים שהסוכן מחובר לשירות ול-Lambda (דרך ה-image/Terraform).
  2. יוצרים משימה (POST /tasks) כדי להפעיל את כל השרשרת.
  3. פותחים X-Ray ובוחרים את ה-trace האחרון.
  4. רואים את ה-service map ואת ה-spans: API → SQS → Lambda — עם הזמן שכל מקטע לקח.
הדיאגרמה קמה לחיים
זו בדיוק הארכיטקטורה מהסילבוס — עכשיו נראית כ-trace אמיתי.
חלק 5

Capstone: הכול ביחד

סוגרים את המעגל — מ-push ועד trace, על המערכת שבניתם.

מה בניתם

המערכת המלאה

Code→ CI→ Docker/ECR→ Terraform→ CD→ EC2 + DynamoDB + SQS/Lambda/SNS→ CloudWatch + X-Ray

שירות Java נקי שרץ בענן, נארז ב-Docker, מתפרס אוטומטית עם CI/CD ו-IaC, שומר נתונים, מעבד אירועים, ומנוטר במלואו — נבנה צעד-אחר-צעד לאורך 8 מפגשים.

זו כל מפת הבנייה
כל 8 השלבים — מלאים. אותו פרויקט אחד, מההתחלה ועד הסוף.
Capstone ריצה מלאה מקצה לקצה
  1. דוחפים שינוי ל-main, וה-CD בונה, דוחף ל-ECR, מריץ terraform apply ומפרס (עם אישור).
  2. מייצרים תעבורה אל ה-API הפרוס — יוצרים כמה משימות.
  3. Logs: מתשאלים את הבקשות ב-Logs Insights.
  4. Metrics: רואים את הגרפים ב-dashboard; מפעילים שגיאה ומקבלים alarm + מייל.
  5. Traces: פותחים ב-X-Ray את השרשרת API → SQS → Lambda.
זהו — סגרתם את המעגל
מ-commit ועד trace, אוטומטית ומנוטר. זה DevOps.
מעבדה + הצגות השלימו את שלב 8
  1. ודאו לוגים מובנים זורמים ל-CloudWatch, ותשאלו ב-Logs Insights.
  2. צרו dashboard + alarm על 5xx → SNS; הפעילו שגיאה ובדקו שההתראה מגיעה.
  3. אמתו trace מלא ב-X-Ray (API → SQS → Lambda).
  4. הציגו לכיתה את המערכת שלכם: push, פריסה, לוגים, מטריקה, ו-trace.
סימן הצלחה
אתם מסוגלים לענות על "האם השירות בריא?", "מה נשבר?", ו"איפה הזמן הלך?" — מהנתונים בלבד.
חובה צ׳קליסט ניקוי עלויות
  1. terraform destroy — מוחק את כל מה שהוקם (EC2, DynamoDB, SQS, SNS, Lambda, ECR...).
  2. בדקו בקונסולה שלא נשארו משאבים דולקים (במיוחד EC2).
  3. בדקו את ה-Budget וההתראות מהמפגש על AWS.
  4. מחקו משאבים ידניים אם נותרו (S3 buckets, log groups).
הכלל החשוב ביותר בענן
מה שמדליקים — מכבים. destroy בסוף, תמיד.
