DevOps: למה אנחנו כאן

לפני שנכתוב קוד — נבין מה הבעיה ש-DevOps פותר, ומה המסע שאפליקציה עוברת מהמחשב שלכם ועד המשתמש.

מושג ליבה

מה זה DevOps?

בפשטות

DevOps הוא הגשר בין כתיבת התוכנה (Development) להפעלתה בעולם האמיתי (Operations). אותו צוות אחראי מהקוד ועד שהוא רץ אצל המשתמש — בעזרת אוטומציה.

לעומק טכני

תרבות + פרקטיקות שמקצרות את ה-lead time (מרעיון לפרודקשן): אינטגרציה רציפה (CI), אספקה רציפה (CD), תשתית כקוד (IaC) וניטור (Observability).

למה אכפת לנו
ארבעת הנושאים של הקורס — AWS, Docker, CI/CD, Observability — הם הכלים שבעזרתם DevOps קורה בפועל, ואנחנו נבנה אותם אחד-אחד על הפרויקט.
הבעיה הקלאסית

"אבל אצלי זה עובד..."

קוד שעובד על מחשב אחד נשבר אצל אחר — או על השרת. למה?

  • גרסאות שונות — אצלי Java 21, על השרת Java 17.
  • תלויות חסרות — ספרייה שמותקנת אצלי ולא שם.
  • הגדרות סביבה — משתני סביבה, פורטים, הרשאות, מערכת הפעלה.
  • "רץ ידנית" — מי זוכר את כל הצעדים?
הפתרון שנלמד

Docker (מפגש 3) אורז קוד + תלויות + סביבה לחבילה אחת שרצה זהה בכל מקום.

ועוד פתרון

CI/CD + IaC הופכים את הצעדים הידניים לסקריפט אוטומטי.

חלק 2

HTTP: איך תוכנה מדברת

כל שירות ענן, כל API, כל אתר — מדברים HTTP. הפרויקט שלנו הוא בדיוק שרת HTTP.

מושג ליבה

מודל Client–Server

Client (דפדפן / curl)— בקשה →Server (ה-API שלנו)
Client← תגובה —Server
בפשטות

הלקוח מבקש, השרת עונה. כמו במסעדה: מזמינים (בקשה), מקבלים צלחת (תגובה).

לעומק טכני

HTTP הוא פרוטוקול request/response מעל TCP. הלקוח פותח חיבור ל-IP+port, שולח בקשה, והשרת מחזיר status, headers ו-body.

אנטומיה

בקשה ותגובה

HTTP Request 
GET /tasks HTTP/1.1
Host: localhost:8080
Accept: application/json
HTTP Response 
HTTP/1.1 200 OK
Content-Type: application/json

{"tasks": [], "count": 0}

שורת בקשה/סטטוס + headers + שורה ריקה + body (כאן JSON).

פעלים

מתודות HTTP

מתודהמשמעותבפרויקט שלנו
GETקריאה / שליפהGET /tasks — רשימת משימות
POSTיצירת משאב חדשPOST /tasks — משימה חדשה
PUTעדכון משאב קייםסימון משימה כבוצעה
DELETEמחיקהמחיקת משימה
לעומק טכני

GET הוא idempotent (קריאה חוזרת = אותה תוצאה). POST אינו — שתי קריאות יוצרות שני משאבים. חשוב כשנבנה תורים ו-retries במפגש 5.

קודי סטטוס

מה השרת עונה

טווחמשמעותדוגמאות

3xxהפניה301, 304
4xxשגיאת הלקוח400, 404 Not Found
5xxשגיאת השרת500, 503
כלל אצבע

4xx = הבקשה שגויה, 5xx = השרת נפל. ההבחנה קריטית לאבחון תקלות במפגש ה-Observability.

Port

IP מזהה מחשב; port מזהה אפליקציה עליו. ה-API שלנו יאזין על 8080 — http://localhost:8080/health.

JSON

{

הדגמה בקשת HTTP ראשונה עם curl

curl שולח בקשות HTTP משורת הפקודה — נשתמש בו לכל אורך הקורס.

  1. בקשת GET: 
    curl https://httpbin.org/get
  2. לראות סטטוס ו-headers עם -i: 
    curl -i https://httpbin.org/get   # השורה הראשונה: HTTP/1.1 200 OK
  3. בקשת POST עם JSON: 
    curl -X POST https://httpbin.org/post \
  -H "Content-Type: application/json" \
  -d '{"title":"Hello"}'
פקודות רב-שורתיות ב-Windows
ה-\ בסוף שורה = המשך-שורה ב-bash (WSL/Git Bash). ב-PowerShell השתמשו ב-backtick בסוף שורה, או כתבו את כל הפקודה בשורה אחת.
חלק 3

Terminal ו-Linux

כל עבודת ענן ו-DevOps עוברת דרך שורת הפקודה. נכיר את הפקודות שילוו אותנו.

מושג ליבה

למה Terminal ולא עכבר?

בפשטות

דרך לתת למחשב פקודות בכתב. נראה מאיים בהתחלה, אבל מהיר, מדויק וניתן לאוטומציה — דברים שאי-אפשר עם עכבר.

לעומק טכני

ה-shell (bash/zsh) מפרש פקודות. שרתי ענן הם Linux ללא ממשק גרפי — הגישה היחידה היא SSH/SSM ל-shell.

חשוב לתלמידי Windows
הדרך הקלה: התקינו WSL2 (Ubuntu) או השתמשו ב-Git Bash — כך כל פקודות ה-Linux בקורס עובדות זהה. לחלופין אפשר PowerShell, ונציג את הפקודות המקבילות היכן שהן שונות.
לתלמידי Windows · התקנה חד-פעמית

מתקינים WSL2 — לינוקס בתוך Windows

המרצה עובד/ת על Mac, וכל הפקודות בקורס הן Linux/Mac. WSL2 נותן לכם Linux אמיתי בתוך Windows, כך שכל פקודה שתראו תעבוד אצלכם בדיוק אותו הדבר. כ‑15 דקות, פעם אחת.

  1. פתחו PowerShell כמנהל — חיפוש "PowerShell", קליק ימני, ואז Run as administrator.
  2. התקינו WSL2 + Ubuntu בפקודה אחת, ואז הפעילו מחדש את המחשב: 
    wsl --install
  3. אחרי האתחול ייפתח חלון Ubuntu — בחרו שם משתמש וסיסמה (זכרו אותם, צריך ל-sudo).
  4. עדכנו את המערכת מתוך חלון ה-Ubuntu: 
    sudo apt update && sudo apt upgrade -y
  5. מכאן עובדים תמיד מחלון ה-Ubuntu — לא מ-PowerShell/CMD. כל פקודות הקורס יעבדו כמו אצל המרצה.
Docker + VS Code
Docker: ב-Docker Desktop, תחת Settings > Resources > WSL Integration, הפעילו עבור Ubuntu.
VS Code: התקינו את התוסף WSL מבית Microsoft (ms-vscode-remote.remote-wsl), ומתוך Ubuntu הריצו code . — העורך והטרמינל ירוצו בלינוקס.
תוספים שימושיים לקורס (בתוך WSL): vscjava.vscode-java-pack · ms-azuretools.vscode-docker · hashicorp.terraform.
שני כללי זהב
שמרו קבצים בתוך לינוקס (~/projects, לא /mnt/c) — מהיר יותר. והריצו פעם אחת git config --global core.autocrlf input כדי שסיומות-שורה לא ישברו סקריפטים.
חלופה קלה (לפקודות בסיסיות)
Git Bash (מגיע עם Git for Windows) מספיק ל-git ולפקודות בסיסיות, אבל WSL2 הוא הקרוב ביותר לסביבת המרצה — מומלץ.

שלושה מושגים

נתיבים, משתני סביבה והרשאות

נתיבים

משתני סביבה

ערכים שהאפליקציה קוראת מהסביבה, למשל APP_PORT. Linux/Mac: export APP_PORT=8080 · PowerShell: $env:APP_PORT="8080".

הרשאות

ב-Linux לכל קובץ הרשאות r/w/x. קובץ מפתח SSH חייב הרשאות מוגבלות (chmod 400 key.pem). ב-Windows: icacls או פשוט מ-WSL/Git Bash — נתקל בזה ב-EC2 במפגש 4.

Git ו-GitHub

בפשטות

Git הוא "מכונת זמן" לקוד: שומר תמונת-מצב (commit) בכל שלב, מאפשר לחזור אחורה ולעבוד במקביל בלי לדרוס.

חלק 5

מה זה בעצם "ענן"?

"הענן" הוא מחשבים של מישהו אחר שאתם שוכרים לפי שימוש. נבין את המודלים והמבנה של AWS — לקראת מפגש 4.

מושג ליבה

On-Prem מול Cloud

בפשטות

פעם: קונים שרת פיזי ומתחזקים אותו. בענן: לוחצים כפתור, מקבלים שרת תוך דקות, משלמים לפי שימוש, ומכבים כשמסיימים.

לעומק טכני

משאבי מחשוב לפי דרישה (on-demand) דרך API, תמחור pay-as-you-go, גמישות (elasticity) ואחריות תפעול משותפת עם הספק.

למה זה משנה לנו
במפגש 4 נריץ את ה-Tasks API על שרת אמיתי בענן תוך דקות — ונכבה אותו בסוף.
מודלי שירות

IaaS · PaaS · SaaS

מודלמה מקבליםאתם מנהליםדוגמה
IaaSתשתית גולמיתמערכת הפעלה ומעלהEC2
PaaSפלטפורמה לקודרק את הקודLambda
SaaSתוכנה מוכנהכלוםGmail
אנלוגיה: פיצה

IaaS = מצרכים ואופים לבד · PaaS = פיצה קפואה לחימום · SaaS = משלוח מוכן. בקורס נתמקד ב-IaaS (EC2) ו-PaaS (Lambda).

המבנה של AWS

Regions ו-Availability Zones

בפשטות

Region = אזור גיאוגרפי (פרנקפורט, אירלנד). AZ = מרכז נתונים בתוך אותו אזור. בוחרים Region קרוב למשתמשים.

לעומק טכני

כל Region מכילה כמה AZs מבודדות פיזית ומחוברות ברשת מהירה. פריסה על כמה AZs = עמידות (high availability). חלק מהשירותים גלובליים (IAM).

טיפ מעשי
תמיד בדקו באיזה Region אתם (פינה עליונה בקונסולה). נעבוד באותו Region לאורך הקורס.

ה-handler של /health

HealthHandler.java 
public void handle(HttpExchange ex) throws IOException {
  String body = "{\"status\":\"UP\"}";            // גוף JSON
  ex.getResponseHeaders().add("Content-Type", "application/json");
  ex.sendResponseHeaders(200, body.getBytes().length);  // סטטוס + אורך
  try (var os = ex.getResponseBody()) { os.write(body.getBytes()); }
}
מה לשים לב

כל handler מקבל HttpExchange שמכיל בקשה ותגובה: קובעים headers, שולחים status, וכותבים body — בדיוק האנטומיה של HTTP.

למה זה חשוב

זו הפשטה (abstraction) שמאפשרת להחליף תשתית בלי לגעת בלוגיקה — נחזור אליה כשנוסיף אחסון אמיתי בענן. דוגמה ל"לבנות היום עם המחר בראש".

interface TaskRepository { List<Task> all(); Task save(Task t); Optional<Task> byId(String id); }
// STORAGE=memory   -> InMemoryTaskRepository   (מפגש 1)
