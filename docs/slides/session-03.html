היום — שלב 3: Package

אורזים את ה-Tasks API ל-Docker image: יחידה אחת שכוללת את הקוד, ה-JDK/JRE וההגדרות — ורצה זהה בכל מקום.

כבר מאחורינו · הבא

מפגשים 1–2: כתבנו, בדקנו, ויש CI ירוק. מפגש 4: ניקח את ה-image הזה ונריץ אותו על שרת אמיתי בענן (AWS EC2).

חלק 1

למה קונטיינרים?

הבעיה ש-Docker פותר — ומה ההבדל בינו לבין מכונה וירטואלית.

מושג ליבה

Container מול VM

בפשטות

VM = מחשב שלם מדומה, עם מערכת הפעלה משלו — כבד ואיטי לעלייה. Container = אריזה של האפליקציה בלבד + מה שהיא צריכה, שחולקת את ה-OS של המארח — קטן ועולה בשניות.

לעומק טכני

VM רץ מעל hypervisor עם guest OS מלא. container הוא תהליך מבודד (namespaces + cgroups) שחולק את ה-kernel של המארח — לכן הוא קל, מהיר וצפוף יותר.

VMContainer
גודלגיגה-בייטיםמגה-בייטים
זמן עלייהדקותשניות
בידודמלא (OS נפרד)ברמת התהליך
הערך

הקונטיינר פותר את "אצלי זה עובד"

ה-image כולל את הכול: הקוד, ה-runtime (JRE), התלויות וההגדרות.

  • אם זה רץ ב-container אצלי — זה ירוץ אותו דבר אצל כל אחד ובכל שרת.
  • אין יותר "התקנת Java נכונה" על כל מכונה — הכול בתוך ה-image.
  • אותו image בדיוק עובר dev → CI → ענן — בלי הפתעות.
המסע של ה-image בקורס
נבנה אותו היום, נבנה אותו גם ב-CI, נדחוף ל-registry, ובמפגש 6 נריץ אותו בענן מ-ECR.
חלק 2

Images, שכבות ו-Registries

אבני הבניין של Docker לפני שנכתוב Dockerfile.

מושג ליבה

Image מול Container

בפשטות

Image = "תבנית" קפואה לקריאה בלבד (כמו קובץ התקנה). Container = מופע רץ של ה-image. מ-image אחד אפשר להריץ הרבה containers.

לעומק טכני

image = שכבות לקריאה-בלבד מוערמות. בהרצה, Docker מוסיף שכבת כתיבה דקה למעלה ומפעיל את התהליך. אנלוגיה: image = class, container = object.

איך נבנה image

שכבות (Layers) ו-Build Cache

בפשטות

כל הוראה ב-Dockerfile יוצרת שכבה. אם שכבה לא השתנתה, Docker משתמש מחדש ב-cache ומדלג עליה — בנייה מהירה הרבה יותר.

לעומק טכני

השכבות ממוזערות (cached) לפי תוכן. סדר ההוראות חשוב: שמים דברים שמשתנים נדירות (תלויות) מוקדם, ודברים שמשתנים הרבה (הקוד) מאוחר — כדי למקסם פגיעות cache.

למה אכפת לנו
ב-CI ובענן בונים images שוב ושוב — cache טוב חוסך דקות יקרות בכל בנייה.
איפה images חיים

Registries

registry הוא "מחסן" ל-images — דוחפים (push) ומושכים (pull), בדיוק כמו Git לקוד.

registryמתי בקורס
Docker Hubהיום — registry ציבורי נוח להתחלה
Amazon ECRמפגש 6 — registry פרטי ב-AWS, ממנו נפרוס לענן
פורמט שם image
user/tasks-api:1.0 — שם משתמש/מאגר:tag. ה-tag מזהה גרסה.
חלק 3

כותבים Dockerfile

המתכון שאומר ל-Docker איך לבנות את ה-image של ה-Tasks API.

הוראות יסוד

ההוראות שצריך להכיר

הוראהמה היא עושה
FROMimage בסיס להתחיל ממנו (למשל Java runtime)
WORKDIRתיקיית העבודה בתוך ה-image
COPYהעתקת קבצים מהפרויקט אל ה-image
RUNהרצת פקודה בזמן הבנייה (למשל התקנה)
EXPOSEתיעוד הפורט שהאפליקציה מאזינה עליו
ENTRYPOINT / CMDהפקודה שרצה כשהקונטיינר עולה
קוד · ה-Dockerfile שלנו

אורזים את ה-JAR

docker/Dockerfile 
FROM eclipse-temurin:17-jre-alpine   # בסיס: Java 17 runtime (רזה)
WORKDIR /app
COPY app/target/*-shaded.jar app.jar   # ה-JAR שבנינו
EXPOSE 8080
ENTRYPOINT ["java","-jar","app.jar"]
שני צעדים היום
קודם בונים את ה-JAR (mvn package), ואז docker build אורז אותו. במפגש 6 נעבור ל-multi-stage build — docker build אחד שבונה גם את ה-JAR וגם מקטין את ה-image.
פקודות הרצה

בנייה והרצה של container

פקודהמה היא עושה
docker build -t name .בונה image בשם name
docker run -p 8080:8080מריץ + ממפה פורט מארח←קונטיינר
-e KEY=valמעביר משתנה סביבה
docker ps / logsקונטיינרים רצים / לוגים
docker exec -it id shכניסה ל-shell בתוך הקונטיינר
Volumes (בקצרה)
קונטיינר הוא חולף — כשמוחקים אותו, מה שנכתב בתוכו נעלם. volume מחבר תיקייה קבועה לשמירת נתונים. אצלנו הנתונים יישמרו בענן (DynamoDB, מפגש 5), לא בקונטיינר.
הדגמה בונים ומריצים את ה-container
  1. גרסת הייחוס של היום: 
    git checkout v3-docker
  2. בונים את ה-JAR ואז את ה-image: 
    mvn -pl app -am -q package
docker build --platform linux/amd64 -f docker/Dockerfile -t tasks-api .
  3. מריצים וממפים פורט: 
    docker run -p 8080:8080 tasks-api
  4. בודקים (חלון שני): 
    curl localhost:8080/health   # {"status":"UP",...}
  5. לוגים וכניסה פנימה: 
    docker ps
docker logs <id>
חלק 4

docker compose

להריץ את האפליקציה (ובהמשך כמה שירותים יחד) בפקודה אחת.

מושג ליבה

מה זה docker compose?

בפשטות

קובץ YAML אחד שמתאר את כל השירותים של הסביבה המקומית, ומריץ את כולם יחד עם docker compose up — בלי לזכור פקודות docker run ארוכות.

לעומק טכני

compose מגדיר שירותים, רשת משותפת ביניהם, פורטים, משתני סביבה ו-volumes — באופן הצהרתי. השירותים מדברים זה עם זה לפי שם השירות.

היום שירות אחד · מפגש 5 שניים
היום נריץ רק את האפליקציה. במפגש 5 נוסיף לאותו קובץ שירות LocalStack (חיקוי AWS מקומי) — וזה בדיוק הכוח של compose.
קוד · compose

קובץ ה-compose שלנו

docker/docker-compose.yml 
services:
  app:
    build:
      context: ..
      dockerfile: docker/Dockerfile
    ports:
      - "8080:8080"
    environment:
      - STORAGE=memory          # במפגש 5 -> dynamodb + localstack

הרצה: docker compose -f docker/docker-compose.yml up

הדגמה מריצים עם compose
  1. מעלים את הסביבה בפקודה אחת: 
    docker compose -f docker/docker-compose.yml up --build
  2. בודקים שהשירות חי: 
    curl localhost:8080/health
  3. מורידים את הסביבה: 
    docker compose -f docker/docker-compose.yml down
שימו לב
docker compose (עם רווח) היא הגרסה הנוכחית (v2), מובנית ב-Docker — לא docker-compose הישן.
חלק 5

דחיפה ל-registry + CI

נשתף את ה-image דרך Docker Hub, ונרחיב את ה-CI שיבנה אותו אוטומטית.

הדגמה דוחפים ל-Docker Hub
  1. התחברות (חד-פעמי): 
    docker login
  2. תיוג ה-image בשם המלא עם tag: 
    docker tag tasks-api <user>/tasks-api:1.0
  3. דחיפה ל-registry: 
    docker push <user>/tasks-api:1.0
  4. עכשיו כל אחד יכול למשוך ולהריץ: docker run <user>/tasks-api:1.0.
קוד · מרחיבים את ה-CI

ה-CI בונה גם את ה-image

מוסיפים צעד ל-job הקיים ממפגש 2 — אחרי הבדיקות, ה-JAR כבר קיים, אז בונים את ה-image:

.github/workflows/ci.yml (תוספת) 
      - name: Build and test
        run: mvn -B verify
      - name: Build Docker image
        run: docker build --platform linux/amd64 -f docker/Dockerfile -t tasks-api:${{ github.sha }} .
בונים, עדיין לא דוחפים
ב-CI רק מוודאים שה-image נבנה. הדחיפה ל-registry בענן (ECR) תתווסף ל-CD במפגש 7.
מעבדה בנו את שלב 3 — 25 דקות
  1. כתבו את docker/Dockerfile (מהשקופית), ובנו: mvn -pl app -am package ואז docker build --platform linux/amd64.
  2. הריצו את הקונטיינר עם -p 8080:8080 ובדקו /health ו-POST /tasks.
  3. בדקו לוגים עם docker logs, ונסו docker exec -it.
  4. הריצו עם compose: docker compose ... up --build.
  5. דחפו את ה-image ל-Docker Hub שלכם.
  6. אתגר: הוסיפו את צעד בניית ה-image ל-ci.yml וראו אותו רץ.
סימן הצלחה
הקונטיינר עונה ב-/health, ה-image נמצא ב-Docker Hub, וה-CI בונה אותו.
משימת בית לפני מפגש 4
  1. הוסיפו .dockerignore (למשל target/, .git) — כדי לא לשלוח קבצים מיותרים ל-build context.
  2. הקטינו את ה-image: בדקו את גודלו (docker images), ונסו base image רזה יותר (למשל alpine/jre מינימלי). השוו גדלים.
  3. ודאו שחשבון ה-AWS (Free Plan) פעיל — נשתמש בו במפגש הבא!
טיזר למפגש 6
ה-multi-stage build ייקח את הקטנת ה-image צעד קדימה — ובלי הצעד הידני של mvn package.
